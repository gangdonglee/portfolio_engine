# 오케스트레이션 작업 지침 (Claude Code 서브에이전트 활용 플레이북)

> 본 문서는 본 프로젝트를 진행하는 동안 **메인 에이전트(Claude)가 어떤 상황에서 어떤 서브에이전트를 어떻게 호출할지** 의 운영 지침이다.
> 코드 작성/리뷰/조사/디버깅 등 일상 작업에서 일관된 패턴을 유지하기 위해 사용한다.

---

## 1. 사용 가능한 서브에이전트

| 이름 | 용도 | 본 프로젝트에서의 활용 |
|---|---|---|
| **Explore** | 코드/심볼/문자열 빠른 검색 (읽기 전용, 발췌 위주) | 기존 DX 샘플, 우리 코드베이스의 특정 함수 위치 찾기 |
| **Plan** | 구현 계획 설계, 트레이드오프 분석 | 새 서브시스템(렌더 그래프, 메모리 얼로케이터 등) 아키텍처 설계 전 |
| **general-purpose** | 멀티스텝 조사, 광범위 검색 | DirectX 12 새 기능 자료 수집, 베스트 프랙티스 비교 |
| **claude-code-guide** | Claude Code/SDK/API 자체 질문 | 본 프로젝트엔 사용 빈도 낮음 |
| **claude** (default) | 그 외 잡다 | 거의 사용 안 함 |

### 1.1 본 프로젝트 커스텀 서브에이전트

`.claude/agents/` 에 정의된 프로젝트 전용 에이전트.

| 이름 | 용도 | 호출 시점 |
|---|---|---|
| **dx12-reviewer** | DX12 코드 리뷰 — 리소스 상태 전이, 펜스/동기화, 디스크립터 힙, 커맨드 리스트 소유권, COM 라이프타임 | 렌더링 관련 .cpp/.h 추가·변경 후 커밋 직전 |
| **oop-reviewer** | C++ 객체지향 원칙 준수 검사 — SRP, 캡슐화, RAII, 의존성 방향, 다형성 적절성 | 새 클래스/모듈 추가 시, 큰 리팩토링 후 |

신규 커스텀 에이전트를 추가할 때는 `.claude/agents/<name>.md` 를 만들고 본 표에 한 줄 추가한다. 정의 파일은 git에 커밋(팀/장기적 활용).

---

## 2. 핵심 원칙

1. **서브에이전트는 도구이지 디폴트가 아님.** 메인 컨텍스트에서 처리 가능하면 그대로 한다. 남용은 cold-start 비용으로 손해.
2. **병렬화 우선.** 독립적 조사·리뷰가 2개 이상이면 한 메시지에서 동시에 띄운다 (직렬 호출은 시간 낭비).
3. **이해를 떠넘기지 않는다.** “이 코드 봐줘”는 금지. 항상 구체적 파일 경로/라인 + 무엇을 봐야 하는지 + 무엇을 반환해야 하는지 명시.
4. **결과 길이 제한을 둔다.** “200단어 이내” 또는 “표 한 장만” 으로 잘라 메인 컨텍스트 보호.
5. **코드 스타일 준수.** 모든 신규/리팩토링 코드는 [docs/CODE_STYLE.md](docs/CODE_STYLE.md) 의 결정을 따른다 — 멤버 prefix `_camelCase`, 타입 alias `int32/uint32`, RAII 생성자, 전방선언, ThrowIfFailed 등. `oop-reviewer` 호출 시 점검 항목에 포함.

## 3. 워크플로우 패턴

### 패턴 A — 병렬 조사 (Research Fan-out)

새로운 시스템 도입 전 자료 조사가 필요할 때.

```
[Main] 결정해야 하는 사안 명세
   │
   ├─► Agent(general-purpose): NVIDIA 공식 가이드 조사
   ├─► Agent(general-purpose): Microsoft DX12 샘플 분석
   └─► Agent(general-purpose): 오픈소스 엔진 사례 (Filament 등)
   │
[Main] 3개 결과 종합 → 결정
```

**예시 사용처**: 디스크립터 힙 관리 전략, 멀티스레드 커맨드 리스트 패턴, 메시 인스턴싱 방식.

### 패턴 B — 독립 리뷰 (Independent Review)

새 파일/서브시스템을 커밋하기 전, 메인이 보지 못한 시각으로 검토.

```
[Main] 코드 작성 완료
   │
   ▼
Agent(general-purpose, code-reviewer 톤):
  "src/render/CommandQueue.cpp 리뷰. 멀티스레드 안전성, 펜스 누수,
   디스크립터 핸들 라이프타임 위주. 200단어 이내 punch list."
   │
   ▼
[Main] 지적 사항 반영
```

**언제**: 새 서브시스템 완성 시, 중요한 리팩토링 후, “자기 검열”이 잘 안 되는 영역(메모리/동기화).

### 패턴 C — 설계 → 검증 (Plan then Validate)

큰 변경 전 Plan 에이전트로 설계안 받고, 별도 에이전트로 비평.

```
[Main] 요구사항
   │
   ▼
Agent(Plan): 구현 계획 안 도출
   │
   ▼
[Main] 계획 검토 → 미심쩍은 부분
   │
   ▼
Agent(general-purpose): "이 계획에서 가장 약한 가정 3개?"
   │
   ▼
[Main] 수정 후 착수
```

**언제**: 렌더 그래프, 잡 시스템, 에셋 파이프라인 등 한 번 잘못 짜면 비용 큰 시스템.

### 패턴 D — 빠른 위치 탐색 (Lookup)

이미 있는 코드의 위치/사용처를 빠르게 찾을 때.

```
Agent(Explore): "FrameContext가 정의된 파일과 사용처 5곳"
```

Grep/Glob로 충분하면 그쪽 우선. Explore는 “여러 라운드 검색 + 종합”이 필요할 때만.

### 패턴 E — 백그라운드 작업 (Background)

장시간이 걸리고 결과가 즉시 필요 없을 때 (`run_in_background: true`).

**용도**:
- 디브로그 초안 작성 (“방금 짠 CommandQueue 설계 결정을 한국어로 300단어 정리”)
- 큰 영역 광범위 조사 (“ozz-animation 라이브러리 전체 API 표면 요약”)

본 코드를 짜는 동안 병렬로 글이 나오게 함.

## 4. 안티 패턴 (하지 말 것)

| 안티패턴 | 왜 나쁜가 | 대안 |
|---|---|---|
| 단순 파일 읽기에 Agent 호출 | cold-start 비용 낭비 | Read 직접 |
| 한 줄 grep에 Explore 호출 | 같은 이유 | Grep 직접 |
| “이 버그 고쳐줘” 식 위임 | 이해 책임 떠넘김. 결과물 신뢰 못함 | 구체 파일/라인 + 무엇이 잘못된 것으로 의심되는지 명시 |
| 결과 길이 제한 없음 | 메인 컨텍스트 오염 | “200단어”, “punch list만” 등 명시 |
| 직렬로 3개 호출 | 시간 3배 낭비 | 한 메시지에서 병렬 호출 |
| 같은 작업 중복 (메인 + 서브) | 토큰 낭비 | 위임했으면 메인은 다른 일 |

## 5. 마일스톤별 오케스트레이션 적용 (예고)

본 프로젝트는 풀스크래치 C++/DX12 엔진이라 단계별로 다음 패턴이 주로 쓰일 예정:

| 단계 | 주요 패턴 | 비고 |
|---|---|---|
| **Foundation** (윈도우/DX12 부트스트랩) | 패턴 D, A | 표준 코드 → 검색 중심 |
| **Renderer Core** (커맨드/디스크립터/리소스) | 패턴 A, C | 설계 결정 잦음 |
| **Asset Pipeline** (메시/텍스처 로딩) | 패턴 A, B | 외부 포맷 조사 + 독립 검증 |
| **Animation** (스켈레톤/스키닝/블렌딩) | 패턴 C, B | 자료구조 결정이 크리티컬 |
| **Gameplay Foundations** (입력/카메라/캐릭터 컨트롤러) | 패턴 D, B | 익숙한 영역, 검색·검토 위주 |
| **Combat** (히트박스/콤보/캔슬) | 패턴 B, E | 폴리시 디테일 많음. 디브로그 병렬 작성 |
| **AI Boss** | 패턴 C, B | 페이즈 설계 → 검증 |
| **Polish & Demo Cut** | 패턴 E | 영상/문서 병렬 작성 |

## 6. 일일 작업 사이클 (권장)

1. **세션 시작**: 어제 작업 결과 요약(메인 컨텍스트). 오늘 목표 1~2개 확정.
2. **조사가 필요하면**: 패턴 A로 병렬 fan-out.
3. **설계가 필요하면**: 패턴 C (Plan → 비평).
4. **코드 작성**: 메인에서 직접. 작은 단위로 빌드 확인.
5. **작업 단위 완료**: 빌드 검증 → 독립 리뷰 (패턴 B, `dx12-reviewer` / `oop-reviewer`) → 지적 사항 반영.
6. **작업 후 커밋 (필수 루틴)**: 의미 있는 작업 단위마다 즉시 커밋한다. 큰 묶음 한 번에 ❌, 작은 단위 자주 ⭕.
   - **단위 기준**: "빌드 가능 + 한 가지 의도된 변경"이면 커밋. 거대 커밋 금지.
   - **메시지 형식**: Conventional Commits 한국어. 예) `feat(render): Device 클래스 추가`, `fix(window): WM_SIZE 시 백버퍼 재생성`, `chore: 빌드 옵션 정리`, `docs(devlog): 02 엔트리 추가`.
   - **커밋 직전 점검**: 무관한 파일 staged 됐는지, 임시 디버그 코드 남았는지, 한글 문서 BOM 있는지.
   - **푸시**: 커밋 후 가능하면 즉시 푸시 (`git push`). 손실 위험 최소화.
7. **단계 완료 시**: `devlog/NN-topic.md` 신규 작성 — 결정·트레이드오프·작업·**실수 빠짐없이**·결과 상세 기록. [devlog/_TEMPLATE.md](devlog/_TEMPLATE.md) 기반. 보통 작업 커밋과는 별도 커밋(`docs(devlog): NN ...`).
8. **세션 종료**: 패턴 E로 추가 정리(디브로그 보강, 다음 세션 to-do 1줄 메모).

## 7. 본 문서 자체의 운영

- 본 문서는 살아 있는 운영 지침이다. 새 패턴이 발견되면 메인 에이전트가 직접 갱신한다.
- 운영 중 발견한 안티 패턴도 §4에 누적한다.
- 너무 길어지면 분할 (별도 `ORCHESTRATION_CASES.md` 등).
